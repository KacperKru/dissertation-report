\section{Specification}\label{sec:spec}
% System design: requirements (what the system must do; optional subsections: functional, non-functional). Research: problem and what is required of a solution.
As discussed in section \ref{sec:related}, it can be observed that each existing device on the market only utilises one input method, whereas the proposed solution needs to listen out for up to three. These methods of input being: terminal commands, physical button interaction and tangible artefact interaction through \gls{nfc} technology.

\subsection{Functional requirements}\label{subsec:funcreq}
To ensure consistency between these modes of input and outlining clear deliverables, the following functional requirements were fufilled:
\begin{itemize}
    \item Pressing the play/pause button \textbf{must} play and pause the current song being played.
    \item Pressing the skip button \textbf{must} skip the current song being played to the next song in the playlist.
    \item The play/pause button \textbf{must} be illuminated on start-up to denote the identity of the function of the button.
    \item The device \textbf{must} be able to be interfaced with remotely via \gls{ssh}.
    \item Sending commands over an \gls{ssh} terminal \textbf{must} perform the action that it has been assigned to do so.
    \item Presenting a \gls{nfc} tag to the \gls{nfc} antenna \textbf{must} perform a playlist switch to the playlist present in the payload of the tag.
    \item The device \textbf{must} connect to a predefined Bluetooth Speaker on start-up.
    \item The device \textbf{must} execute the main script on start-up, avoiding unnecessary remote management. 
\end{itemize}
\subsection{Non-functional requirements}\label{subsec:nonfuncreq}
In a similar vein, the proposed solution needs to function in an efficient and safe manner. It is unreasonable to expect the end users to have to wait extended periods of time for actions to be performed, and potentially compromise their network with the proposed solution as the attacker's entry point. Therefore, the following non-functional requirements were also fulfilled:
\begin{itemize}
    \item Commands over the \gls{ssh} terminal should take \textbf{no longer than} 5 seconds to execute.
    \item Bluetooth should connect to a predetermined Bluetooth speaker in \textbf{no more than} 10 attempts.
    \item Users without authorisation should \textbf{not} be able to connect to the device over \gls{ssh}.
    \item The device needs to be \textbf{intuitive} and easy to use.
    \item Start-up of the device should take \textbf{no longer than} 2 minutes.
    \item The \gls{nfc} antenna on the device should read a presented \gls{nfc} tag successfully in \textbf{no more than} 10 attempts.
    \item The device should be \textbf{scalable}, and not suffer any performance drops as the playlists increase.
    \item The device should \textbf{reboot} after a power cut and return to a \textbf{workable state} once power is restored.
\end{itemize}

\section{System design}\label{sec:design}
% System design: architecture, data flow, behaviour (UML, ER, state charts). Use diagrams. Research: use ``Approach and solution'' for your proposed solution(s).
The system has been designed around three main hardware components:
\begin{enumerate}
    \item \texttt{Raspberry Pi 4 Model B}
    \item \texttt{Arduino Expansion Shield} acting as an Arduino Leonardo board
    \item \texttt{Bluetooth Speaker} in this case, a Google Nest Mini.
\end{enumerate}
However, the Raspberry Pi and Arduino Board are connected unconventionally, due to a GrovePi+ being already attached to the header pins of the Raspberry Pi. They are instead connected through the USB ports on the Raspberry Pi itself. The Arduino is necessary here to connect the \gls{nfc} Reader Module, as its Base Shield possesses a UART connection, that the GrovePi+ does not have access to. A detailed diagram of how these components are connected with all accompanying sensors, buttons and speakers can be seen in Figure \ref{fig:sysoverH}.

The software this project consists of can be separated into 3 main scripts:
\begin{enumerate}

    \item \texttt{server.py}, hosing the core functionality of the media player and receiving commands from both the Arduino and client.py.
    \item \texttt{reading\_nfc.ino}, used for reading the data off \gls{nfc} tags and sending them into the serial port.
\end{enumerate}
An additional script for the Arduino also exists called "\texttt{writing\_nfc.ino}" but it is only necessary when creating new \gls{nfc} tags for new or existing playlists. A detailed diagram of how these interact with each other can be seen in Figure \ref{fig:sysoverS}.

The Python scrips only utlises 2 external libraries to execute all of its functions, the rest being included in a Python 3 install. The list of the libraries and what they were for is as follows:
\begin{itemize}
    \item grovepi, one of the external libraries that allows for direct interaction with the pins on the GrovePi+ shield that has the buttons connected to it.
    \item python-vlc, the other external library that allows for the music to be played through Raspbian's default media player VLC. Controls for the songs are managed through this library.
    \item time, native Python library that introduced delays in the scripts, specifically to allow for Bluetooth connection to occur without disruption.
    \item threading, native Python library that allowed for multiple functions (in this case, reading \gls{nfc} tags and receiving terminal commands through a socket) to run concurrently.
    \item socket, native Python library that utilises BSD Socket API to communicate between \texttt{server.py} and \texttt{client.py}.
    \item subprocess, native Python library that allowed for execution of terminal commands (in this case, bluetoothctl).
    \item os, native Python library that read all of the song files in a given directory to construct a playlist.
    \item serial, native Python library that facilitated reading from the serial port. 
    \item re, native Python library that could enact regular expressions on the payloads of the \gls{nfc} tags.
    \item random, native Python library that facilitated the "shuffle" functionality in the media player.
\end{itemize}

The C++ sketches for the Arduino utilised a library provided by Seeed Studios to interface with the \gls{nfc} reader, using \gls{ndef} technology \citep{SeeedStudios2020}.

\section{Ethical considerations}\label{sec:ethics}
Dealing with a protected characteristic such as disability means ethics is intertwined with this project. To be able to user test with the stakeholders, full ethics approval is required. However, Given the timescale of this project, this was not feasible. 

The author instead opted to omit meeting the stakeholders directly, and in their stead, work with carers in the field instead. An argument can be made that since the author's intention was to communicate with private care homes with the assistance of ENRICH Cymru, simplified ethics could still be applied as these are not NHS staff.

For the sake of simplicity, the decision to include workers was also scrapped. Had more time and resources be allocated towards this project, researchers in the field would instead be the subject of user tests to hopefully provide actionable feedback on the proposed solution.

\subsection{Button Interaction}\label{subsec:btnint}
The proposed solution's main form of interaction is that of button presses on the Raspberry Pi. Once the GrovePi module was installed, development commenced on recognising the button presses.

An issue arose when trying to address the button presses. The first LED button was connected to the D2 port on the GrovePi shield and the other was connected to the D7 port. These ports were chosen as they were directly opposite each other on the shield, making it easier to distinguish between them both. Algorithm \ref{alg:badBtnTest} was used to test these ports.
\begin{algorithm}[!h]
    \caption{Unsuccessful Button Port Test - Python}
    \label{alg:badBtnTest}
    \begin{lstlisting}[language=Python]
    import time
    import grovepi

    buttonP_pin = 2
    buttonS_pin = 7

    grovepi.pinMode(buttonP_pin, "INPUT")
    grovepi.pinMode(buttonS_pin, "INPUT")

    print("Ready!")

    buttonP_was_pressed = False
    buttonS_was_pressed = False

    try:
        while True:
            buttonP_state = grovepi.digitalRead(buttonP_pin)
            buttonS_state = grovepi.digitalRead(buttonS_pin)
            if buttonP_state == 0 and not buttonP_was_pressed:
                print("Button Play pressed!")
                buttonP_was_pressed = True

            elif buttonP_state == 1 and buttonP_was_pressed:
                print("Button Play released!")
                buttonP_was_pressed = False
                
            if buttonS_state == 0 and not buttonS_was_pressed:
                print("Button Skip pressed!")
                buttonS_was_pressed = True

            elif buttonS_state == 1 and buttonS_was_pressed:
                print("Button Skip released!")
                buttonS_was_pressed = False

            time.sleep(0.05)

    except KeyboardInterrupt:
        print("\nExiting.")
    \end{lstlisting}
\end{algorithm}
\newpage
Algorithm \ref{alg:badBtnTest} was not printing any statements on button presses and the logic seemed sound. testing different port numbers, \texttt{buttonP\_pin} and \texttt{buttonS\_pin} were routinely swapped out to their corresponding ports, until they were returned to the ports they were originally connected to (2 and 7). The algorithm's pin values were not changed as an oversight but output was being registered when the \texttt{buttonP} was connected to D2 but and the value of \texttt{buttonP\_pin} was 3. An \textbf{offset of 1} was identified in the port numbering.

The amended line can be seen in Algorithm \ref{alg:goodBtnTest}.
\begin{algorithm}[!h]

        buttonS_pin = 8

To adjust for scalability and the implementation of more than one playlist, the function to load the playlist was moved into a Python function. This was done for modularity and avoiding repeating code when a playlist is to be switched. It takes two arguments, the playlist name and if it is to be shuffled. If shuffle is not passed, it defaults to False. The code snippet can be seen in Algorithm \ref{alg:loadplaylist}.
\begin{algorithm}[!t]
  \caption{load\_playlist() - Python}
  \label{alg:loadplaylist}
  \begin{lstlisting}[language=Python]
    def loadplaylist(playlistName, shuffle="false"):
        if playlistName == '': # Handling null inputs
            return False
        
        try:
            os.listdir(f'AudioTracks/{playlistName}')
        except FileNotFoundError: # If playlist folder doesn't exist
            return False
        
        playlist = instance.media_list_new()
        
        if shuffle.lower() == "true":
            songs = os.listdir(f'AudioTracks/{playlistName}')
            random.shuffle(songs)
            for sound in songs:
                print(sound)
                media = instance.media_new(f'AudioTracks/{playlistName}/{sound}')
                playlist.add_media(media)
        else:
            for sound in sorted(os.listdir(f'AudioTracks/{playlistName}')): # Sorts the playlist in alphabetical order
                print(sound)
                media = instance.media_new(f'AudioTracks/{playlistName}/{sound}')
                playlist.add_media(media)
        media_player.set_media_list(playlist)
        return True
  \end{lstlisting}
\end{algorithm}

The proposed solution can now play songs based off a pre-determined order in a folder, or play it shuffled.
\subsection{Terminal Interaction}\label{subsec:terminal}
Remote Management was one of the key aims for this project and without the presence of a GUI to facilite this requirement, terminal commands were introduced. The "remote" aspect came from connecting to the device using \gls{ssh} and interfacing with the terminal provided from the connection. Should the user opt for a desktop environment, \gls{vnc} was also available, with the user having to open a terminal window through the desktop environment.

This was originally setup through the use of a \texttt{While True} loop executing in tandem with the button recognition commands, using the \texttt{threading} library, as seen in Algorithm \ref{alg:threadingEG}. The button recognition was moved to a function named "background" to enable threading to occur, whereas the code within the function is the same as Algorithm \ref{alg:mappedBtnPress}.

Once \texttt{main.py} was executed, it awaited inputs and utilised an "If" statement to decide on what to do, as seen in Algorithm \ref{alg:IFstatement}. The author was to make use of a Structural Pattern Match in the form of a "match case" statement, but this feature wasn't implemented till Python 3.10 (one version ahead of the one present on the Raspberry Pi) \citep{PythonSoftwareFoundation2023}.

\begin{algorithm}[!h]
  \caption{Running Terminal Commands using \texttt{threading}}
  \label{alg:threadingEG}
  \begin{lstlisting}[language=Python]
    threading1 = threading.Thread(target=background)
    threading1.daemon = True
    threading1.start()
  \end{lstlisting}
\end{algorithm}

\begin{algorithm}[!h]
  \caption{If Statements for Terminal Commands}
  \label{alg:IFstatement}
  \begin{lstlisting}[language=Python]
    while True:
        command_input = input("Enter a command: ")
     
        if command_input.lower() == "p":
            media_player.pause()
        elif command_input.lower() == "s":
            media_player.next()
        elif command_input.lower() == "switch":
            playlistChange = input("What's the name of the playlist?: ")
            try:
                media_player.stop()
                loadplaylist(playlistChange)
                print("Switching Playlist...")
                time.sleep(0.5)
                media_player.play()
            except FileNotFoundError:
                print("Playlist doesn't exist")
                media_player.play()
        elif command_input.lower() == "help":
            print('--------------\nhelp: displays commands available\ns: skips current song\np: pauses current song\nexit: closes music program\n--------------')
        elif command_input.lower() == "exit":
            print("\nExiting")
            grovepi.digitalWrite(buttonP_pin -1, 0)
            break 
        else:
            print('Invalid command, try again. To see list of commands, type "help"')
  \end{lstlisting}
\end{algorithm}

The commands the user has at their disposal are:
\begin{itemize}
    \item play/pause, represented as "p"
    \item skip, represented as "s"
    \item switch, allowing the user to switch to the playlist passed as an argument
    \item help, proving a menu on what each command does 
\end{itemize}

This initial approach however, wasn't viable the moment the main.py was executed at start time, covered fully in section \ref{subsec:startup}. No terminal was available when the Python file was executed as part of a service so alternative methods of passing commands had to be explored. Anthropic's \gls{llm} Claude was able to assist here, suggesting the utilisation of sockets \citep{AnthropicMarch2026}. Instead of an open terminal running the terminal through \texttt{main.py}, a secondary smaller script \texttt{client.py} would send a message for \texttt{main.py} (now being renamed \texttt{client.py}) to then decode and execute the command the message contained. The socket receiver now also needs to run in tandem, so it becamse the 2\textsuperscript{nd} thread and is executed using the \texttt{threading} module.

Both \texttt{client.py} and \texttt{server.py} use the \texttt{socket} module to send and receive these messages. \texttt{client.py} can be seen in Algorithm \ref{alg:clientpy} and \texttt{server.py}'s socket receiver function can be seen as a snippet in Algorithm \ref{alg:socketReceiverSnip}. One limitation of \texttt{client.py} is the ability to only run one command per Python script execution; sending more caused the tunnel to collapse and would need to be restarted regardless.

\begin{algorithm}[!h]
  \caption{client.py - Python}
  \label{alg:clientpy}
  \begin{lstlisting}[language=Python]
    import socket
    import sys

    def send_command(command):
        if command == '': # Handle Null Values
            return "Invalid command\n"
        
        socket_path = '/tmp/musicplayer.sock'
        client = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM) # Using a UNIX socket
        client.connect(socket_path)
        
        client.send(command.encode())
        response = client.recv(1024).decode() # Receives a maximum of 1024 bytes
        client.close()
        return response

    if __name__ == '__main__': # Prevent running as script
        command = ' '.join(sys.argv[1:])
        response = send_command(command)
        print(response)
  \end{lstlisting}
\end{algorithm}

\subsection{Start-up Behaviour}\label{subsec:startup}
The device should not need remote interaction on start-up. If this was to be omitted, this would overcomplicate the device and make it inferior to devices already on the market, where simplicity thrives. Remote Management should be an add-on, rather than a necessary tool. The device therefore needed to start playing music the moment it turns on, or reboots after a power cut. This was done through the use of \texttt{systemd} on the Raspberry Pi, where a service was created that would run \texttt{server.py} the moment a user has logged on the Pi, which should happen automatically as the Raspberry Pi has been configured to sign in automatically. Algorithm \ref{alg:playerServ} shows how \texttt{mediaplayer.service} file utilised in \texttt{systemd}. This service needs to then be moved to \texttt{/etc/systemd/system/} activated using \texttt{systemctl start mediaplayer.service} and will be executed the next time the Raspberry Pi reboots. 
\begin{algorithm}[!h]
  \caption{mediaplayer.service}
  \label{alg:playerServ}
  \begin{lstlisting}
    [Unit]
    Description=Music Player
    After=network.target sound.target

    [Service]
    ExecStart=/usr/bin/python3 /home/pi/cm3203-project/server.py
    WorkingDirectory=/home/pi/cm3203-project
    User=pi
    # This helps VLC and PulseAudio find the user session bus (as only created when user has logged in)
    Environment=XDG_RUNTIME_DIR=/run/user/1000
    Restart=on-failure
    RestartSec=5

    [Install]
    WantedBy=multi-user.target
  \end{lstlisting}
\end{algorithm}

As the device utilises a Bluetooth speaker to play the desired media, the Bluetooth device needs to be paired on start-up as well. This is ensured through a function in \texttt{server.py} that executes before the main script does. It attempts to connect to a given MAC address a total of 10 times, before it times out. This time-out then allows for support to be carried out by a carer to connect a Bluetooth device manually. Algorithm \ref{alg:bluetoothCon} demonstrates how Bluetooth connection is executed.
\begin{algorithm}[!h]
  \caption{connect\_bluetooth() - Python}
  \label{alg:bluetoothCon}
  \begin{lstlisting}[language=Python]
    def connect_bluetooth(mac_address):
        time.sleep(10) # Buffer time to allow for main start-up process to occur
        print('Attempting initial connection...')
        output = subprocess.run(["bluetoothctl", "connect", mac_address], capture_output=True, text=True)
        for i in range(10):
            if 'Connection successful\n' not in output.stdout:
                print('Attempting reconnection...')
                output = subprocess.run(["bluetoothctl", "connect", mac_address], capture_output=True, text=True)
                print(output.stdout)
                time.sleep(3)
  \end{lstlisting}
\end{algorithm}

\subsection{NFC Reading and Command Execution}\label{subsec:NFC}
As time allowed during the development of this project, tangible item interaction utilising \gls{nfc} was implemented alongside the button interaction. Due to the \gls{nfc} module communicating over a \gls{uart} interface, an Arudino was necessary to facilite this communication. The provided \gls{nfc} module could be switched to interface using \gls{i2c} but soldering would be necessary to switch interfaces \citep{gunengyu2022} and an official Python library from the manufacturer does not exist.

The Arduino could not be fitted to the top of the Raspberry Pi due to the GrovePi+ already being seated on top, having the devices communicate over a USB bus instead. The Arduino was fitted with a base shield that contained a \gls{uart} that the \gls{nfc} module was attached to. The Arduino had two sketches, one for writing to the \gls{nfc} tags (Algorithm \ref{alg:writeNFC}) and one for reading them (Algorithm \ref{alg:readNFC}). Both scripts were adapted from sketches provided from the IOT Garage Lab Book GitLab \citep{Perera2024} and included a library to be able to interface with the module on the Arduino \citep{SeeedStudios2020}.

\begin{algorithm}[!h]
  \caption{writing\_nfc.ino - C++}
  \label{alg:writeNFC}
  \begin{lstlisting}[language=C++]
    #include <NfcAdapter.h>
    #include <PN532/PN532_HSU/PN532_HSU.h>

    PN532_HSU pn532hsu(Serial1);
    NfcAdapter nfc(pn532hsu);

    String record_entry = "NAME_OF_PLAYLIST";

    void setup() {
        Serial.begin(9600);
        while(!Serial);
        Serial.println("NDEF Writer");
        nfc.begin();
    }

    void loop() {
        if (nfc.tagPresent()) {
            Serial.println("NFC TAG FOUND");
            // 14. Create an NdefMessage object to write a simple text or URI record to the NFC tag.
            NdefMessage message = NdefMessage();
            message.addUriRecord(record_entry);
            // 15. Attempt to write this new message to the NFC tag using nfc.write().
            bool success = nfc.write(message);
            
            if (success) {
                Serial.println("Write succeeded.");
            } else {
                // NOTE: The original code had a capitalization error ("SERIAL.println").
                //       Ensuring consistency here as "Serial.println".
                Serial.println("Write failed.");
            }
        }
    }
  \end{lstlisting}
\end{algorithm}

\begin{algorithm}[!h]
  \caption{reading\_nfc.ino - C++}
  \label{alg:readNFC}
  \begin{lstlisting}[language=C++]
    #include <NfcAdapter.h>
    #include <PN532/PN532_HSU/PN532_HSU.h>

    PN532_HSU pn532hsu(Serial1);
    NfcAdapter nfc(pn532hsu);

    void setup() {
        Serial.begin(9600);
        while(!Serial);
        Serial.println("NDEF Reader");
        nfc.begin();
    }

    void loop() {
        if (nfc.tagPresent()) {
            NfcTag tag = nfc.read();
            tag.print(); // Prints to serial port
        }
        delay(1000);
    }
  \end{lstlisting}
\end{algorithm}

Once the tags have been written and been presented to the \gls{nfc} module, \texttt{server.py} extracts the information using Regular Expressions from the serial output of the USB bus, located through executing \texttt{ls} in \texttt{/dev/} \citep{Quibits2026}, and switches playlists based off the payload passed on the \gls{nfc} tag, as seen in Algorithm \ref{alg:readNFCPy}.

\begin{algorithm}[!h]
  \caption{nfc\_reader\_switch() - Python}
  \label{alg:readNFCPy}
  \begin{lstlisting}[language=Python]
    #Accepting NFC Tags
    serial_port = '/dev/ttyACM0'
    baud_rate = 9600

    ser = serial.Serial(serial_port, baud_rate) # Opening Serial Port on USB port Arduino is connected to.
    
    while True:
        serData = ser.readline().decode('utf-8', errors='ignore')
        if serData.startswith("    Payload"): # Selects specifically Payload
            if serData.startswith("    Payload L") == False: # Ignores Payload Length
                print(serData)
                message = re.findall(r'\.(\w+)', serData) # Extracts just text information from Payload
                media_player.stop()
                if len(message) == 0:
                    print('Error reading NFC tag, attempt again...')
                else:
                    loadplaylist(message[0])
                media_player.play()
  \end{lstlisting}
\end{algorithm}
The device can be interfaced with \gls{nfc} tags.
